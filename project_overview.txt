## 1. PROJECT OVERVIEW

### **Problem Statement**
Traditional inventory management systems struggle with:
- Manual stock tracking leading to errors
- Lack of real-time inventory visibility
- Difficulty in identifying low-stock items
- Poor transaction tracking and audit trails
- Inefficient supplier management
- Limited reporting and analytics capabilities

### **Proposed Solution**
A web-based inventory management system using:
- **MongoDB** for flexible, scalable data storage
- **Node.js/Express** for robust backend API
- **REST API** architecture for system integration
- **Real-world dataset** (UCI Online Retail) for realistic demonstrations

### **Project Scope**
**In Scope:**
- Product catalog management
- Category and supplier management
- Stock tracking and transaction logging
- Low stock alerts and notifications
- Reporting and analytics dashboard
- Data import from UCI dataset
- RESTful API development

**Out of Scope:**
- User authentication (can be added as enhancement)
- Payment processing
- Barcode scanning integration
- Mobile application
- Email notifications

### **4.2 MongoDB Schema Design**

#### **Why MongoDB?**
1. **Flexible Schema**: Easy to modify structure as requirements evolve
2. **Document-Based**: Natural fit for product catalogs (nested data)
3. **Scalability**: Horizontal scaling for large datasets
4. **JSON Format**: Easy integration with JavaScript/Node.js
5. **Performance**: Fast reads for inventory queries
6. **Embedded Documents**: Store addresses, specifications within documents

#### **Collections Design**

**Products Collection:**
```javascript
{
  _id: ObjectId("..."),
  name: "Dell Wireless Mouse",
  sku: "MOUSE-001",
  description: "Wireless optical mouse with USB receiver",
  category: ObjectId("..."), // Reference to Category
  supplier: ObjectId("..."), // Reference to Supplier
  quantity: 150,
  reorderLevel: 20,
  price: 850,
  cost: 600,
```

### **4.3 Database Normalization**

**Normalization Level**: 3NF (Third Normal Form)

**Advantages:**
- No data redundancy (category name stored once)
- Easy to update (change supplier info in one place)
- Data integrity (referential integrity via ObjectId)
- Efficient storage

**Design Decisions:**
1. **Referenced vs Embedded**: Used references for frequently changing data (categories, suppliers)
2. **Indexed Fields**: sku (unique index), category, supplier
3. **Virtual Fields**: Calculated fields like `isLowStock` computed on-the-fly

### **5.2 MVC Architecture**

**Model**: Mongoose schemas defining data structure  
**View**: JSON responses (no traditional view in REST API)  
**Controller**: Business logic processing requests  

### **5.3 API Design (RESTful)**

#### **Design Principles**
1. **Resource-Based URLs**: `/api/products` not `/api/getProducts`
2. **HTTP Methods**: GET (read), POST (create), PUT (update), DELETE (delete)
3. **Status Codes**: 200 (success), 201 (created), 404 (not found), 500 (error)
4. **JSON Format**: All requests and responses in JSON
5. **Versioning**: `/api/v1/...` for future compatibility

#### **API Endpoints Documentation**

**Products Endpoints:**
```
GET    /api/products              - Get all products
GET    /api/products/low-stock    - Get low stock products
GET    /api/products/:id          - Get single product
POST   /api/products              - Create new product
PUT    /api/products/:id          - Update product
DELETE /api/products/:id          - Delete product
```

**Categories Endpoints:**
```
GET    /api/categories            - Get all categories
GET    /api/categories/:id        - Get single category
POST   /api/categories            - Create category
PUT    /api/categories/:id        - Update category
DELETE /api/categories/:id        - Delete category
```

**Similar structure for Suppliers and Transactions**

## 7. SYSTEM FEATURES

### **7.1 Core Features Implementation**

#### **Feature 1: Product Management**

**What it does:**
- Add new products to inventory
- View all products with details
- Update product information (price, quantity, etc.)
- Delete products from system
- Search products by name, SKU, category

**Business Logic:**
```
When adding a product:
1. Validate required fields (name, SKU, price)
2. Check if SKU already exists (must be unique)
3. Verify category and supplier exist
4. Set default values (status=active, quantity=0)
5. Create audit log entry
6. Save to database
7. Return success response with product ID
```

**Demo Scenario:**
"Faculty, I'll now demonstrate adding a new product called 'Wireless Headphones'. The system validates the SKU uniqueness, ensures the price is positive, and automatically timestamps the entry."

#### **Feature 2: Low Stock Alert System**

**What it does:**
- Automatically identifies products below reorder level
- Displays list of low-stock items
- Shows recommended reorder quantity
- Provides category-wise breakdown

**Business Logic:**
```
Low Stock Detection:
1. Query all products where quantity <= reorderLevel
2. Calculate deficit: reorderLevel - currentQuantity
3. Sort by deficit (most critical first)
4. Group by category for insights
5. Calculate total value of restocking needed
```

**Calculation Example:**
```
Product: Dell Mouse
Current Quantity: 8
Reorder Level: 20
Status: LOW STOCK
Recommended Order: 12 units
Estimated Cost: 12 Ã— â‚¹600 = â‚¹7,200
```

**Demo Scenario:**
"The system automatically monitors stock levels. Here you can see 15 products currently below their reorder points, with the total restocking cost calculated at â‚¹2.5 lakhs."

#### **Feature 3: Transaction Logging**

**What it does:**
- Records every stock movement
- Maintains complete audit trail
- Supports three transaction types:
  - **IN**: Stock received (purchases, returns from customers)
  - **OUT**: Stock dispatched (sales, damages)
  - **ADJUSTMENT**: Stock corrections (inventory count differences)

**Business Logic:**
```
When recording a transaction:
1. Validate product exists
2. Check quantity is positive
3. For OUT: Ensure sufficient stock available
4. Update product quantity:
   - IN: quantity += transaction_qty
   - OUT: quantity -= transaction_qty
   - ADJUSTMENT: quantity = new_value
5. Log transaction with timestamp
6. Create reference to invoice/PO
7. Record user who made change (if auth enabled)
```

**Demo Scenario:**
"Every movement is logged. This transaction shows 50 units received from supplier on Nov 3rd, referenced to Purchase Order PO-2024-156. The system automatically updated the product quantity from 25 to 75."

#### **Feature 4: Category-wise Analysis**

**What it does:**
- Groups inventory by categories
- Shows total stock value per category
- Identifies top-performing categories
- Displays category-wise distribution

**Metrics Calculated:**
```javascript
For each category:
- Total Products: Count of products
- Total Quantity: Sum of all quantities
- Total Value: Sum (quantity Ã— price)
- Average Price: Average of product prices
- Low Stock Items: Count of products below reorder level
- Stock Percentage: Category stock / Total stock Ã— 100
```

**Demo Scenario:**

#### **Feature 5: Supplier Performance Tracking**

**What it does:**
- Lists all registered suppliers
- Shows products supplied by each
- Tracks supplier reliability
- Displays contact information

**Metrics:**
```
Per Supplier:
- Total Products Supplied: Count
- Total Stock Value: Sum of product values
- Active Products: Products currently in stock
- Last Delivery Date: Most recent transaction
```
